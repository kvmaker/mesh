# A05. 公网 direct / NAT traversal — 详细分析

## 实施记录

（实施后填写：分支、改动文件、关键设计、测试结果）

## 1. 背景与动机

Tailscale、ZeroTier 都把公网直连作为主要目标，relay 仅作为 fallback。当前 Mesh 所有 client-to-client 公网流量都依赖 server 中继。

目标是在 UDP hole punching 可行时直连，不可行时回落到 relay。

## 2. 当前项目状态

- 没有 UDP listener 用于数据面。
- 没有 NAT traversal 探测逻辑。
- 没有 endpoint discovery 协议。

## 3. 实施方案

### 3.1 Endpoint 注册

每个 client 在控制面注册时附带：

```text
{
  "udp_endpoint": "x.x.x.x:51820",
  "nat_type": "cone|unknown"  // 可选 STUN
}
```

### 3.2 Hole punching 流程

参考 Tailscale 的“coordination layer makes both sides send packets at the same time”策略：

```text
1 A 询问 control plane：目标 B 的 udp_endpoint
2 A 同时收到 B：目标 A 的 udp_endpoint
3 A 与 B 几乎同时向对方发送 udp probe
4 双方 NAT 看到 outbound 后允许 return flow
5 probe 成功 → direct path 生效
```

### 3.3 PathManager 集成

- 新增 `direct` path 类型。
- 失败熔断与定期重试。

### 3.4 Symmetric NAT 处理

对称 NAT、企业网难以打洞。策略：

- 多次重试；
- 失败时回退到 relay；
- 上报 control plane 标记 `relay_only`。

### 3.5 与 E2E 加密协同

- direct 通道应叠加 A10 的 session key，确保 server 不能解密。
- 端到端加密与 NAT traversal 握手解耦。

## 4. 测试计划

- 模拟：双 cone NAT 直连成功。
- 模拟：symmetric NAT 回退到 relay。
- 端到端：跨国链路直接 vs relay 性能对比。
- 长期：direct 路径稳定性（保活、连接迁移）。
