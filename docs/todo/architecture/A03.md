## 1. 背景与动机

Tailscale Peer Relays 允许用户把网络内连通性好的节点作为专用中继，路径优先级为 direct > peer relay > DERP。对于跨境、跨运营商场景特别有效。

Mesh 当前所有中继都走 server，没有利用网络内可能的优质中转节点（如家里 NAS、公司跳板机）。

## 2. 当前项目状态

- 设备表 / 设备配置没有 relay 角色字段。
- 转发只支持 server 中继。
- 没有 peer relay 概念。

## 3. 实施方案

### 3.1 角色与配置

设备表增加：

```text
relay_enabled    bool
relay_priority   int
```

客户端配置：

```yaml
relay:
  allow_peer_relay: true
  preferred_peer_relays:
    - home-nas
    - office-gateway
```

### 3.2 路径选择

`PathManager` 增加 `peer_relay` 类型：

```text
优先级：
1. direct
2. preferred peer relay
3. 任意 peer relay
4. server relay
```

### 3.3 中继节点

relay 节点收到需要中继的包时按相同规则查 `PathManager`，发送给真正目的 peer。

### 3.4 安全

- relay 节点不应该作为开放代理。
- ACL 限制可被 relay 的流量。
- 中继节点默认 deny，避免成为跳板。

## 4. 测试计划

- 单测：peer relay 路径选择。
- 集成测试：3 节点 A-B-C，A 走 B 中继到 C。
- 失败模拟：peer relay 不可用时回退到 server。
