## 1. 背景与动机

Mesh 的关键价值是“把包送到对端”，但当前架构把所有“如何送到”压成一种方式：client → server → client。参考 Tailscale、ZeroTier 的设计，至少有以下几种路径是常见的：

- direct（LAN 或公网直连）
- peer relay（网络内节点作为中继）
- server relay（中心中继）
- multi-lane relay（多连接 / 多 protocol 通道）

如果不先把“路径”抽象出来，每加一种路径就会扩散到 client/server 的 read loop，难以演进。

## 2. 当前项目状态

- `internal/tunnel/router.go` 中只有 `Router.routes[netip.Addr] → ClientConn`。
- 没有 path 概念，没有 health、queue depth、failure tracking。

## 3. 实施方案

### 3.1 数据结构

```go
type PathType string

const (
    PathRelay     PathType = "relay"
    PathPeerRelay PathType = "peer_relay"
    PathDirect    PathType = "direct"
)

type Path struct {
    Type       PathType
    Endpoint   string
    RTT        time.Duration
    QueueDepth int
    Healthy    bool
    FailCount  int
    LastUsed   time.Time
}

type PeerRoute struct {
    MeshIP netip.Addr
    Paths  []*Path
    Lock   sync.RWMutex
}

type PathManager struct {
    mu     sync.RWMutex
    routes map[netip.Addr]*PeerRoute
}
```

### 3.2 选择策略

```text
1 过滤 unhealthy
2 按 RTT 升序
3 queue depth 过高降权
4 连续失败熔断
5 周期性探测恢复
```

### 3.3 接口

```go
func (pm *PathManager) RegisterRoute(meshIP netip.Addr, paths ...*Path)
func (pm *PathManager) Select(meshIP netip.Addr) (*Path, error)
func (pm *PathManager) ReportResult(p *Path, err error, latency time.Duration)
func (pm *PathManager) Snapshot() map[netip.Addr][]Path
```

### 3.4 集成

- `internal/tunnel/client.go`: 转发时 `pm.Select(dst) → Send`。
- `internal/tunnel/router.go`: 替换为 `PathManager`。
- API/CLI: 暴露当前 path 状态。

## 4. 测试计划

- 单测：连续失败后熔断；定期探测恢复。
