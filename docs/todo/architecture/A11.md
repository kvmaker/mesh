## 1. 背景与动机

当前数据面完全基于 WebSocket over TCP，内层 TCP 流量（如 SSH、文件传输、HTTP）会形成 TCP-over-TCP，引入队头阻塞与重传放大。Cloudflare WARP 的方案是同时支持 WireGuard 和 MASQUE（基于 QUIC）。Tailscale 也在持续改进 NAT traversal 与连接迁移能力。

QUIC 的优势包括：

- 基于 UDP，规避外层 TCP 队头阻塞。
- 0-RTT/1-RTT 握手加速。
- 内建 connection migration，IP/NAT 切换不需要重连。
- 与 E2E 加密（A10）结合后可以承载 opaque tunnel payload。

但 QUIC 也带来新挑战：UDP 在某些企业网、跨国线路下可能被 QoS 限速或屏蔽，需要 fallback。

## 2. 当前项目状态

- 数据通道：仅 WebSocket/TLS（`internal/tunnel/client.go` 中的 `websocket.Dial`、`Conn.Read/Write`）。
- 没有 transport 抽象概念。
- 没有连接迁移支持。

## 3. 实施方案

### 3.1 Transport 抽象

```go
type Transport interface {
    Dial(ctx context.Context, peer Peer) (Conn, error)
    Accept(ctx context.Context) (Conn, error)
    Type() string
}

type Conn interface {
    Send(pkt []byte) error
    Recv(ctx context.Context) ([]byte, error)
    Close() error
    RTT() time.Duration
}
```

两个实现：

- `WSTransport`: 现有 WebSocket/TLS，作为 fallback。
- `QUICTransport`: 新增 QUIC 实现，作为 fast path。

### 3.2 协议握手

在 client/server 握手阶段协商 transport：

```text
client hello:
  - 设备 ID
  - 公钥
  - 支持的 transport: [quic, ws-tls]

server ack:
  - 设备信息
  - 选定 transport: quic
  - session id
```

如果 QUIC 不可达（UDP 屏蔽），回落到 `ws-tls` 并把降级结果报告给控制面。

### 3.3 QUIC 配置

- 库：quic-go（Go 生态主流）。
- 0-RTT：根据安全策略默认关闭或基于 device 信任开启。
- 连接迁移：启用，但需要重写 endpoint 时正确处理 dst 切换。
- MTU：避免 IP fragmentation，QUIC max packet size 控制在合理范围。
- 与 A10 协同：QUIC 提供 transport 安全，A10 仍负责 app 层 E2E 加密。

### 3.4 Fallback 策略

```text
优先 QUIC
  ├─ 握手失败 / 心跳失败 → 切换到 ws-tls
  └─ 长期失败 → 上报 control plane，标记设备为 ws-tls-only
```

## 4. 测试计划

- 单测：transport 抽象接口 mock。
- 集成测试：QUIC 握手、0-RTT、连接迁移。
- 端到端测试：
  - 跨国/高 RTT 链路下吞吐对比 ws-tls。
  - 内层 TCP 拥塞场景下延迟对比。
  - 移动端切网/WiFi 切换后连接保持。
- 失败模拟：UDP 屏蔽下自动回落到 ws-tls。

## 6. 上线策略

- 默认仍走 WebSocket/TLS；QUIC 通过 client profile 灰度开启。
- 通过可观测性收集 QUIC vs WS 的 RTT/吞吐数据，再决定默认切换。
- 移动端谨慎开启 0-RTT，避免重放风险。

## 7. 风险

- UDP 限速 / 屏蔽：必须 fallback，否则大面积不可用。
- QUIC 库复杂度：高版本升级可能引入破坏性变更。
- MTU 调优：跨网环境差异大。
- 安全：QUIC 仅是 transport security，业务仍需要 A10 的 E2E 加密。
