## 1. 背景与动机

P02 引入有界 SendQueue 后，如果队列没有边界，内存会无限增长；如果没有明确丢包策略，延迟会失控。这是典型 VPN 转发反模式：包堆在内存里，几秒后才发出去，拥塞反而放大。

## 2. 当前项目状态

- `internal/tunnel/router.go` 的 `ClientConn` 目前没有队列、没有 drop 统计。
- 还没有 `queue_depth` 之类的可观测性。

## 3. 实施方案

### 3.1 队列配置

```go
type SendQueueConfig struct {
    Size         int
    DropPolicy   DropPolicy // drop_tail / drop_head
}
```

默认：

```text
Size:       1024 或 4096
DropPolicy: drop_tail（实现简单）
```

低延迟偏好：

```text
DropPolicy: drop_head（保留新包，丢弃旧包）
```

### 3.2 行为

```text
enqueue(pkt):
    if queue.full:
        switch policy:
        case drop_tail:
            drop_count++
            return ErrQueueFull
        case drop_head:
            drop_count++
            discard queue.head
            push queue.tail
```

### 3.3 统计

```go
type ClientConn struct {
    ...
    DropPackets    atomic.Uint64
    QueueDepth     atomic.Int64
    QueueMaxDepth  atomic.Int64
}
```

## 4. 测试计划
