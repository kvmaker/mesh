## 1. 背景与动机

当前 Mesh 采用单 server 星型中继。这个模型简单可靠，但所有 client-to-client 流量都受单 gateway 的地理位置、带宽和可用性影响。

多 gateway 的目标不是立刻实现 P2P，而是在仍然使用 relay 的前提下，让 client 能连接多个中转节点，并为每个 peer 选择更健康的路径。

## 2. 当前项目状态

当前配置中 client 只保存：

- `ServerDomain`
- `DeviceSecret`
- `DeviceIP`
- `NetworkCIDR`

转发路由表是：

```text
mesh IP → ClientConn
```

这意味着一个目的地址只有一个中继路径，没有 path 选择空间。

## 3. 实施方案

### 3.1 引入路径抽象

```text
PeerRoute
  ├─ MeshIP
  ├─ Paths[]
  └─ SelectedPath

Path
  ├─ Type: relay/direct
  ├─ GatewayID
  ├─ LaneID
  ├─ RTT
  ├─ QueueDepth
  ├─ Healthy
  └─ FailCount
```

MVP 中只实现 `relay` path，但保留 `direct` 类型，为未来 P2P 做准备。

### 3.2 多 gateway 连接

client 配置从单 server 扩展为：

```json
{
  "server_endpoints": [
    "gateway-a.example.com",
    "gateway-b.example.com"
  ]
}
```

client 同时维护多个 gateway session。

### 3.3 路径选择策略

默认策略：

```text
1 过滤 unhealthy path
2 优先 RTT 低
3 queue_depth 过高则降权
4 连续失败则熔断
5 定期探测恢复
```

### 3.4 MVP 限制

为避免 gateway 互联复杂度，MVP 假设：

```text
所有 client 都连接同一组 gateway；每个 gateway 都能直接转发任意在线 client。
```

## 4. 测试计划

- 单测：PathManager 选择最低 RTT healthy path。
- 单测：queue_depth 高时降权。
- 单测：path 连续失败后熔断。
- 集成测试：一个 gateway 不可用时切换到另一个 gateway。
