## 1. 背景与动机

TUN 库（`golang.zx2c4.com/wireguard/tun`）支持 batch read/write，但当前实现只传长度为 1 的 `bufs` 和 `sizes` 数组（`internal/tunnel/client.go:140-142`）。高包量场景下，单次循环只处理 1 个包，频繁的 `Read/Write` 调用增加 syscall 与调度开销。

## 2. 当前项目状态

- `internal/tunnel/client.go` 的 TUN read loop：`bufs = make([][]byte, 1)`。
- `internal/tunnel/server.go` 的 `readTUN` 同样使用长度为 1 的 buffer。
- 两者都按 1 包/次处理。

## 3. 实施方案

### 3.1 配置

```go
tun_batch_size: 16 (默认)
```

### 3.2 流程

```text
loop:
    n, err = TUN.Read(bufs[0:batchSize], sizes, offset)
    for i in 0..n:
        pkt = bufs[i][offset : offset+sizes[i]]
        route(pkt)
```

### 3.3 注意事项

- macOS utun 的 batch 行为需要实测；如果表现与单包无差异，配置可保持 1。
- batch 太大可能增加单轮处理时间，引入延迟抖动；先保守取 16。

## 4. 测试计划

- 单测：mock TUN 返回多包时处理顺序正确。
- 集成测试：macOS / Linux 分别验证 batch 行为。
- 基准测试：与当前单包处理对比 RTT/吞吐。
