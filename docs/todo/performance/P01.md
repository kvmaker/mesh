## 1. 背景与动机

本机 ping 自己的 mesh IP（`10.100.0.2`）会命中 TUN 路由（例如 `10.100/24 utun7`），client 把包发给 server，server 又按路由表发回同一个 client，路径如下：

```text
A TUN → A mesh client → server → A mesh client → A TUN（回包同样走两轮）
```

实测中 `ping 1.14.193.183 ≈ 10ms` 但 `ping 10.100.0.2 ≈ 20ms+`，正是这种发夹中继的代价。

## 2. 当前项目状态

- `internal/tunnel/client.go`: client TUN → WS 转发不判断 `dst == localIP`。
- `internal/tunnel/packet.go`: 已经提供 `ExtractDstIP`，可以复用。
- server 视角：`internal/tunnel/server.go` 收到 `dst == 10.100.0.2` 时在路由表里能查到本机 client，会再次发出。

## 3. 实施方案

在 client 的 TUN read loop 中：

```text
pkt = read TUN
dst = ExtractDstIP(pkt)
if dst == localIP:
    drop or write back to TUN（处理选项见下）
    record debug counter
    continue
enqueue(pkt) as before
```

### 处理选项

- 选项 A：直接丢弃，并计数。MVP 推荐。
- 选项 B：写回 TUN，让内核处理。需要小心避免形成循环；macOS utun 行为需要实测。

### README 说明

README 增加：

```text
不要用 ping 自己 mesh IP 作为延迟基准；该路径是本地地址发夹，
仅用于验证 mesh 启动，应使用 ping 10.100.0.1（server）或
ping 10.100.0.x（其他 client）。
```

## 4. 测试计划

- 单测：TUN 读到 `dst == localIP` 时不入队。
- 集成测试：client 启动后本机 ping 其他 client 仍正常。
- 人工测试：观察本机 mesh IP 行为符合预期。
