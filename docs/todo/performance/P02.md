## 1. 背景与动机

当前 server/client 的转发热路径中，读循环会直接执行写操作。对于 VPN 转发来说，这是一个典型风险点：慢连接、WebSocket 写阻塞、对端网络抖动都可能反向阻塞读循环，导致包处理延迟放大。

更稳妥的模型是“读只负责收包和路由，写由每个连接自己的 writer goroutine 串行完成”。这样每个连接有明确的写入所有权，避免并发写同一个 WebSocket，也方便加入背压、丢包、批处理和多 lane。

## 3. 实施方案

### 3.1 新增发送队列

在 `ClientConn` 中增加：

```go
SendQueue chan Packet
Done      chan struct{}
```

`Packet` 封装原始 IP 包：

```go
type Packet struct {
    Data []byte
}
```

MVP 阶段可以先复制包数据，避免生命周期复杂化；P06 再引入 buffer pool。

### 3.2 单写者 writeLoop

每个 WebSocket 连接建立后启动一个 `writeLoop`：

```text
1 从 SendQueue 读取 packet
2 调用 Conn.Write 写入 WebSocket
3 更新 Tx 统计
4 遇到写错误时关闭连接或取消上下文
```

### 3.3 转发路径改为 enqueue

server：

- `clientReadLoop`: `dst` 命中其他 client 时，调用 `dest.Enqueue(pkt)`。
- `routePacket`: server TUN 读到包后也调用 `cc.Enqueue(pkt)`。

client：

- 可以先保持现状；后续可把 TUN 写、WS 写也统一为队列模型。

## 4. 测试计划

- 单测：队列未满时 `Enqueue` 成功。
- 单测：队列满时返回明确错误或 drop 计数增加。
- 单测：`writeLoop` 从队列取包并调用 mock writer。
- 集成测试：两个 client 的 ping 仍可通过。
- race 检查：确保没有并发写同一个 WebSocket。
