## 1. 背景与动机

当前协议假设一个 WebSocket Binary Message 就是一个原始 IP 包。这虽然简单，但限制：

- 难以做包级批处理（对 ping、TCP ACK、DNS 等小包收益明显）；
- 难以承载控制帧（path probe、ping、pong、版本协商）；
- 难以承载元数据（lane id、gateway id）；
- 一旦未来加 direct / peer relay 路径，控制信令需要单独通道。

## 2. 当前项目状态

- `internal/tunnel/client.go`: `conn.Write(ctx, websocket.MessageBinary, pkt)`。
- `internal/tunnel/server.go`: 同样直接以原始 IP 包作为 WS 消息。

## 3. 实施方案

### 3.1 Frame 格式

```text
+--------+--------+----------+
| version| type   | payload  |
+--------+--------+----------+
```

类型：

```go
const (
    FrameIPPacket = 1
    FrameBatch    = 2
    FramePing     = 3
    FramePong     = 4
    FrameControl  = 5
)
```

### 3.2 Batch 编码

```text
FrameBatch payload:
  +--------+----------+----------+-----+
  | count  | len1     | packet1  | ... |
  +--------+----------+----------+-----+
```

### 3.3 兼容旧协议

```text
if firstByte >> 4 in {4, 6}:
    treat as raw IPv4/IPv6 packet (old protocol)
else:
    decode new frame
```

由于 IPv4 首字节常为 `0x45`，IPv6 常为 `0x60`，冲突概率低。

### 3.4 批处理策略

- 攒包条件：max batch packets = 16，max batch bytes = 64KB，max wait = 1ms。
- 超过 1ms 即使未满也立即 flush，避免延迟放大。
