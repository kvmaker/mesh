## 1. 背景与动机

当前 server 转发热路径中每个 IP 包都会打印一行日志，例 `internal/tunnel/server.go:179` 的 `log.Printf("packet: len=%d dst=%s", ...)`。在 VPN 高包量场景下，这个日志会带来明显的：

- 格式化开销；
- 锁与 I/O 开销；
- systemd / journald 压力；
- 延迟抖动（写 stderr 阻塞）。

日志对排障是必要的，但默认不应该在热路径上每包打印。

## 2. 当前项目状态

关键文件：

- `internal/tunnel/server.go`：每个包都有 debug 级别日志。
- `internal/tunnel/client.go`：TUN/WS 主循环中没有日志，但仍受默认 logger 行为影响。

## 3. 实施方案

- 引入 debug 开关：
  - 环境变量 `MESH_DEBUG_PACKET=1`。
  - 或配置文件 `debug.packet_log`。
- 热路径仅在 debug 开启时打印。
- 周期性聚合日志（默认开启）：每 N 秒输出统计：
  ```text
  [stats] pkts=... bytes=... drops=... route_miss=...
  ```
- 错误日志（写失败、连接关闭、TUN 错误）保留原样。

**伪代码**

```text
if debug.packet_log:
    log.Printf("packet: ...")
else:
    stats.observePacket(len(pkt), dst)
```

## 4. 测试计划

- 单元/人工：debug 关闭时打 N 包，无 packet log。
- debug 开启时按预期打印。
- 周期性聚合日志频率符合配置。
