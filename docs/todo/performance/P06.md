## 1. 背景与动机

当前每收到一个 TUN 包都会 `make([]byte, sizes[0])` 并 `copy`（`internal/tunnel/client.go:162-163`）。VPN 高包量场景下，频繁分配短生命周期切片会增加 GC 压力，尾延迟抖动明显。

## 2. 当前项目状态

- `internal/tunnel/client.go`: read 后 `make` + `copy`，写回 TUN/WS 后 GC。
- `internal/tunnel/server.go`: `routePacket` 同样 `make` + `copy` + `Write`。

## 3. 实施方案

### 3.1 PacketBuffer 设计

```go
type Packet struct {
    Buf     []byte
    Len     int
    Release func()
}
```

`Release` 是归还到 pool 的回调。

### 3.2 Pool

```go
var packetPool = sync.Pool{
    New: func() any {
        b := make([]byte, maxMTU+100)
        return &b
    },
}
```

### 3.3 生命周期

```text
TUN.Read() →
  buf = pool.Get()
  copy into buf[:n]
  pkt = Packet{Buf: buf, Len: n, Release: func(){ pool.Put(buf) }}
  enqueue(pkt)

writeLoop →
  ws.Write(pkt.Buf[:pkt.Len])
  pkt.Release()
```

### 3.4 注意事项

- 严禁 enqueue 后立刻 Release。
- writer 写完前 buf 不能复用。
- 必须有 release-once 机制或显式使用 `defer Release()`，避免异步路径中 release 时机错误。
- race detector 测试必须开启。

## 4. 测试计划

- 单测：Release 调用次数与 packet 数一致。
- race 测试：并发 enqueue/写不会数据串包。
- 基准测试：对比分配次数与延迟分布。
