## 1. 背景与动机

一个 client 与 server 之间只有单条 WebSocket 数据连接，所有流量共享一条外层 TCP。一旦某流写阻塞或出现单连接队头阻塞，所有流量都会被拖慢。多 lane（多条并行 WebSocket）可以分散风险，但必须按内层 flow 固定选择 lane，避免 TCP 乱序。

## 3. 实施方案

### 3.1 抽象

```go
type Lane struct {
    Conn      *websocket.Conn
    SendQueue chan Packet
    ID        int
    RTT       atomic.Int64
    Load      atomic.Int64
}

type PeerSession struct {
    DeviceID string
    IP       netip.Addr
    Lanes    []*Lane
}
```

### 3.2 Lane 选择

按内层五元组 hash：

```text
hash(srcIP, dstIP, proto, srcPort, dstPort) % laneCount
```

保证同一 flow 固定 lane，避免乱序。

ICMP 用 `src,dst,icmp_id` 选取。

### 3.3 建立 lane

- control lane 1 条，用于 keepalive、path probe。
- data lane N 条（默认 2-4），按需扩展。

### 3.4 协议扩展

与 P07 frame 协议协调：

```text
frame header: { version, type, lane_id, ... }
```

或单独注册时携带 `lane_id` 标识身份。

## 4. 测试计划

- 单测：flow hash 选 lane 稳定性。
- 集成测试：同一 TCP flow 跨 lane 不会出现乱序。
- 故障模拟：单 lane 异常时不影响其他 lane。
